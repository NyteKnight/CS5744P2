Test Plan for Get-A-room Online Hotel Booking System

In this document, we will outline our high level test plan for the get-a-room Online Hotel Booking System. This system is designed to allow users to locate available hotel rooms in a desired area and manage the entirety of the booking process. Our test plan will seek to ensure the functionality of this system from an integration and validation perspective.

Part 1: Test Scope

To properly test this plan, we will need to approach the system from the requirements described by the team for their final product. The requirements are shown below in Table 1 below, which was provided by the get-a-room team.

Functional Requirement

#### Description
FR-1 Users have the ability to create an online account. Email address, password, and name are required to create an account.
FR-2 Authentication requires email address and password.
FR-3 Users are able to complete booking requests as a guest or with a saved account.
FR-4 Ability to view and manage previous and future bookings and itineraries will be available to users with a saved account.
FR-5 Bookings can be canceled by accessing a saved account or searching by confirmation number.
FR-6 Check-in/checkout-out dates, number of guests, number of rooms, and type of room can be edited after a booking is completed, if available at the selected hotel.
FR-7 Users may be required to provide a form of payment at the time of booking; UI must be able to accept payment information. User receives an error message if payment fails.
FR-8 Confirmation page with a confirmation number will be displayed once booking is completed.
FR-9 Confirmation email will be sent to the user email address with confirmation number once booking is completed.
FR-10 The following search criteria are required to search for available hotel rooms: Address (full address, city, or zip code), check-in and check-out dates, number of rooms, number of adults and children.
FR-11 The following search criteria are optional: Distance from location entered, price range, star rating of hotel, guest rating of hotel.
FR-12 If a saved account is being used, search results will be ranked based on machine learning algorithms.
FR-13 Machine learning engine utilizes the following inputs based on the user’s previous hotel choices: distance from location entered, price range, star rating of hotel, guest rating of hotel. Engine will generate ranked results based on the user’s preferences.
FR-14 Search results can be manually filtered or sorted by the user based on: distance from location entered, price high-to-low, price low-to-high, star rating of hotel, guest rating of hotel.

Non-Functional Requirements

#### Description
NFR-1 Get-a-room will support a website and mobile apps for Android and iOS devices.
NFR-2 Search engine and booking processing service must interface with multiple hotel APIs to retrieve search results.
NFR-3 Payment processing service must interface with multiple financial institution APIs to process payments.
NFR-4 Cloud database solutions will be utilized.
NFR-5 Databases must have capability to scale up or down if/when necessary.
NFR-6 User personal data and payment information must be secured.
NFR-5 Website and mobile application must be available 99.99% of the time.

Table 1. System requirements described by the get-a-room team.

The objective of our testing will be to evaluate the design’s functional, performance, and internal design characteristics. The requirement identifiers labeled in Table 1 will help to describe the testing plan as it encompasses each characteristic of the system. This process is further described in the section that follows.

Functional Characteristics

Requirements tested: FR-1 - FR-14

Our test plan will need to evaluate whether or not the system is able to complete all tasks laid out in Table 1. Functionally, it will also need to be assessed to ensure all modules are able to integrate with one another. It will also need to be tested for correctness through validation testing. Due to the relative simplicity of the application, our test process will cover all of the Functional Requirements stated by the development team. The intricate details of the machine learning algorithm that produces suggestions based upon user interaction will be outside of the test scope. However, our test plan will cover the validity of the products of this algorithm.

Internal Design Characteristics

Requirements tested: NFR-1 - NFR-4

Non-Functional requirements 1-4 cover the internal design of the application. We intend to test these by giving a pass or fail rating based on the ability of the design to accomplish these tasks. NFR-1 and NFR-4 will be satisfied if the design does fulfill these requirements. We will consider NFR-2 and NFR-3 complete if the application can interface with 3 hotel and banking institutions respectively

Performance Characteristics

Requirements tested: NFR-5 - NFR-7

The performance requirements specified by the developers cover the scalability, integrity, and availability of the application. Our test scope will cover the investigation of how these requirements are satisfied. Since a database storage solution is used on the server side, NFR-5 will be satisfied simply based on the solution chosen. NFR-6 will be assessed as a part of validation testing, and NFR-7 will be tested by simulating a heavy transaction load and monitoring the system’s response.
