Part 2: Test Plan - Integration Testing

To integrate the three modules (client side, server side, and orchestrator), the best way is through three phases of continuous integration. Continuous integration allows for developers to integrate code into the repository frequently. Each integration is also automatically built and automatically tested.

Each phase consists of coding, automated testing, reporting, and source control. Each of these phases will be tested in isolation. This is to test the basic functionality of each module, detect early and easy to track bugs, and eliminate the possibility of errors being caused by another module’s code. After code is committed into the repository, it will be automatically tested. Reports will determine if integrated code has passed the units tests. If not, integrated code will be source controlled and will restart the at the beginning of the phase of coding. If tests are passed, that phase will be completed and the module will be deployed.

Phase 1: Client Side Testing

Phase one contains the client side portion of the system. This includes the user interface, system management, and core functionality of the web application. The user interface is the only way the end users will interact with the get a room web application. Testing must be done so that all features on the front end are correctly displaying and properly functioning. The system management needs to be tested on how well the user interface can handle authentication of users and guests. This portion of the testing will be cursory in this section as we are simply trying to verify that the system works as a whole, but more thoroughly examined during validation testing. Core functionality such as itinerary management and confirmation allows for a meaningful user experience.We believe that testing the client side system in Phase one will allow us to gain insight into features users like and don’t like earlier in the build process which will prevent us from having us to double back and retest portions of the code late in the integration assessment process.

Phase 2: Server Side Testing

Phase two focuses on the server side portion of the application, the database. Since the database will hold a large collection of data, testing must be thorough and intensive to assure the correctness of information. We will also handle testing the scalability of the database in this portion. As the creators do not wish to have any physical servers and instead rely on the Cloud for server side processing, we do not need to perform any testing on the server side system beyond this verification.

Phase 3: Orchestrator Testing

The final phase is bringing in the system that connects the client and server, the orchestrator. Correct configuration will be necessary to ensure the interoperability of the modules. To ensure this, payment and reservation processing will be part of the final phase. Payment processing will be tested by making HTTP requests to financial company APIs and handling the HTTP responses. Testing is to make sure the web application is able to generate revenue and profit. This also applies to hotel HTTP web service requests so the orchestrator has the most up to date hotel information.
