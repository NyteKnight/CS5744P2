Test Plan for Get-A-room Online Hotel Booking System

In this document, we will outline our high level test plan for the get-a-room Online Hotel Booking System. This system is designed to allow users to locate available hotel rooms in a desired area and manage the entirety of the booking process. Our test plan will seek to ensure the functionality of this system from an integration and validation perspective.

Part 1: Test Scope

To properly test this plan, we will need to approach the system from the requirements described by the team for their final product. The requirements are shown below in Table 1 below, which was provided by the get-a-room team.

Functional Requirement

#### Description
FR-1 Users have the ability to create an online account. Email address, password, and name are required to create an account.
FR-2 Authentication requires email address and password.
FR-3 Users are able to complete booking requests as a guest or with a saved account.
FR-4 Ability to view and manage previous and future bookings and itineraries will be available to users with a saved account.
FR-5 Bookings can be canceled by accessing a saved account or searching by confirmation number.
FR-6 Check-in/checkout-out dates, number of guests, number of rooms, and type of room can be edited after a booking is completed, if available at the selected hotel.
FR-7 Users may be required to provide a form of payment at the time of booking; UI must be able to accept payment information. User receives an error message if payment fails.
FR-8 Confirmation page with a confirmation number will be displayed once booking is completed.
FR-9 Confirmation email will be sent to the user email address with confirmation number once booking is completed.
FR-10 The following search criteria are required to search for available hotel rooms: Address (full address, city, or zip code), check-in and check-out dates, number of rooms, number of adults and children.
FR-11 The following search criteria are optional: Distance from location entered, price range, star rating of hotel, guest rating of hotel.
FR-12 If a saved account is being used, search results will be ranked based on machine learning algorithms.
FR-13 Machine learning engine utilizes the following inputs based on the user’s previous hotel choices: distance from location entered, price range, star rating of hotel, guest rating of hotel. Engine will generate ranked results based on the user’s preferences.
FR-14 Search results can be manually filtered or sorted by the user based on: distance from location entered, price high-to-low, price low-to-high, star rating of hotel, guest rating of hotel.

Non-Functional Requirements

#### Description
NFR-1 Get-a-room will support a website and mobile apps for Android and iOS devices.
NFR-2 Search engine and booking processing service must interface with multiple hotel APIs to retrieve search results.
NFR-3 Payment processing service must interface with multiple financial institution APIs to process payments.
NFR-4 Cloud database solutions will be utilized.
NFR-5 Databases must have capability to scale up or down if/when necessary.
NFR-6 User personal data and payment information must be secured.
NFR-5 Website and mobile application must be available 99.99% of the time.

Table 1. System requirements described by the get-a-room team.

The objective of our testing will be to evaluate the design’s functional, performance, and internal design characteristics. The requirement identifiers labeled in Table 1 will help to describe the testing plan as it encompasses each characteristic of the system. This process is further described in the section that follows.

Functional Characteristics

Requirements tested: FR-1 - FR-14

Our test plan will need to evaluate whether or not the system is able to complete all tasks laid out in Table 1. Functionally, it will also need to be assessed to ensure all modules are able to integrate with one another. It will also need to be tested for correctness through validation testing. Due to the relative simplicity of the application, our test process will cover all of the Functional Requirements stated by the development team. The intricate details of the machine learning algorithm that produces suggestions based upon user interaction will be outside of the test scope. However, our test plan will cover the validity of the products of this algorithm.

Internal Design Characteristics

Requirements tested: NFR-1 - NFR-4

Non-Functional requirements 1-4 cover the internal design of the application. We intend to test these by giving a pass or fail rating based on the ability of the design to accomplish these tasks. NFR-1 and NFR-4 will be satisfied if the design does fulfill these requirements. We will consider NFR-2 and NFR-3 complete if the application can interface with 3 hotel and banking institutions respectively

Performance Characteristics

Requirements tested: NFR-5 - NFR-7

The performance requirements specified by the developers cover the scalability, integrity, and availability of the application. Our test scope will cover the investigation of how these requirements are satisfied. Since a database storage solution is used on the server side, NFR-5 will be satisfied simply based on the solution chosen. NFR-6 will be assessed as a part of validation testing, and NFR-7 will be tested by simulating a heavy transaction load and monitoring the system’s response.

Part 2: Test Plan - Integration Testing

To integrate the three modules (client side, server side, and orchestrator), the best way is through three phases of continuous integration. Continuous integration allows for developers to integrate code into the repository frequently. Each integration is also automatically built and automatically tested.

Each phase consists of coding, automated testing, reporting, and source control. Each of these phases will be tested in isolation. This is to test the basic functionality of each module, detect early and easy to track bugs, and eliminate the possibility of errors being caused by another module’s code. After code is committed into the repository, it will be automatically tested. Reports will determine if integrated code has passed the units tests. If not, integrated code will be source controlled and will restart the at the beginning of the phase of coding. If tests are passed, that phase will be completed and the module will be deployed.

Phase 1: Client Side Testing

Phase one contains the client side portion of the system. This includes the user interface, system management, and core functionality of the web application. The user interface is the only way the end users will interact with the get a room web application. Testing must be done so that all features on the front end are correctly displaying and properly functioning. The system management needs to be tested on how well the user interface can handle authentication of users and guests. This portion of the testing will be cursory in this section as we are simply trying to verify that the system works as a whole, but more thoroughly examined during validation testing. Core functionality such as itinerary management and confirmation allows for a meaningful user experience.We believe that testing the client side system in Phase one will allow us to gain insight into features users like and don’t like earlier in the build process which will prevent us from having us to double back and retest portions of the code late in the integration assessment process.

Phase 2: Server Side Testing

Phase two focuses on the server side portion of the application, the database. Since the database will hold a large collection of data, testing must be thorough and intensive to assure the correctness of information. We will also handle testing the scalability of the database in this portion. As the creators do not wish to have any physical servers and instead rely on the Cloud for server side processing, we do not need to perform any testing on the server side system beyond this verification.

Phase 3: Orchestrator Testing

The final phase is bringing in the system that connects the client and server, the orchestrator. Correct configuration will be necessary to ensure the interoperability of the modules. To ensure this, payment and reservation processing will be part of the final phase. Payment processing will be tested by making HTTP requests to financial company APIs and handling the HTTP responses. Testing is to make sure the web application is able to generate revenue and profit. This also applies to hotel HTTP web service requests so the orchestrator has the most up to date hotel information.

Part 3: Test Procedure for Build

  1. Client Side System
    1. User Interface
      1. Test Objective: FR-1 Create online account
        1. Technique
          1. Access get a room web application on internet browser
          2. Create account on web application
        2. Completion Criteria: Account creation validation
      2. Test Objective: FR-3 Complete book requests as guest or member
        1. Technique
          1. Attempt to book requests as a guest
          2. Validate correct requests are displayed
          3. Attempt to book requests as a member
          4. Validate correct requests as displayed
        2. Completion Criteria: Client displays the correct data
    2. System Management
      1. Test Objective: FR-2 Authentication requires email address and password
        1. Technique
          1. Log into get a room web application using email address and password
          2. Validate authentication
        2. Completion Criteria: Invoke authentication using valid/invalid data
    3. Core Function
      1. Test Objective: FR-3, FR-4, FR-5, FR-6 Manage bookings and itineraries for user accounts
        1. Technique
          1. Book future itineraries
          2. Assign past itineraries to user test account
          3. Cancel or edit future itineraries
          4. Validate itinerary manager displays bookings
        2. Completion Criteria: Validate web application functions as designed
      2. Test Objective: FR-8, FR-9 Test web application confirmation response functionality
        1. Technique
          1. Book itinerary
          2. Pay with valid payment type
          3. CValidate confirmation responses from web application
        2. Completion Criteria: Validate client side confirmation response
      3. Test Objective: FR-10 to FR-14 Test search functionality
        1. Technique
          1. Enter valid/invalid data into search function
          2. Validate search results
        2. Completion Criteria: Validate client side search results
      4. Test Objective: NFR-1 Verify functionality of website, Android, iOS apps
        1. Technique
          1. Replicate all client side functional requirements across all deployment platforms
        2. Completion Criteria: Verify ability of clients to fulfill requirements
  2. Server Side System
    1. Database
      1. Test Objective: Validate correctness of database queries
        1. Perform a set of database queries (CREATE, READ, UPDATE, and DELETE)
        2. Validate returned results are correct
      2. Completion Criteria: Validate the correctness of the collection of data and the structure of the database schema
      3. Test Objective: NFR-5 Test the scalability of the database
        1. Technique
          1. Programatically add a large amount of user information to the database
          2. Programmatically add a large amount of reservation data to the database
          3. Remove all but a very small amount of data from the database
        2. Completion Criteria: Validate database scalability
  3. Orchestrator
    1. Payment Processing
      1. Test Objective: FR-7, NFR-3 Invoke server side application payment transactions
        1. Technique
          1. Pay with valid/invalid payment types
          2. Validate pay transaction response
          3. Simulate a large amount of concurrent transactions
          4. Verify ability to interface with at least 3 financial institutions
        2. Completion Criteria: Validate server side financial transaction processing between orchestrator and financial institution APIs
    2. Reservation Processing
      1. Test Objective: NFR-2 Invoke server side HTTP requests between hotel APIs
        1. Technique
          1. Make a series of HTTP requests to hotel APIs
          2. Validate and confirm HTTP requests perform as designed
          3. Verify ability to interface with at least 3 hotel platforms
        2. Completion Criteria: Validate server side hotel reservation processing between orchestrator and hotel APIs
